## 1. 项目概述

### 1.1 项目目标

本阶段目标不是先做复杂的 AI 功能或命令系统，而是先完成一个可上线的个人博客网站：

- 网站定位为个人博客，服务于作者本人持续输出观点文章、方法论和价值观。
- 首版优先完成站点结构、页面承载、桌面端显示和移动端自适应。
- 为后续 AI 集成预留内容结构和页面空间，但本期不实现 AI 能力本身。
- 最终通过 GitHub 账号完成 GitHub Pages 部署，并在真实 Chrome 中完成验收。

### 1.2 当前代码现状

- 当前仓库只有一个静态入口页面 [index.html](/Users/zfc/code/fak111.github.io/index.html:1)，没有框架、构建系统或多页面结构。
- 当前页面是单篇文章展示页，内容和样式全部内联在 [index.html](/Users/zfc/code/fak111.github.io/index.html:8) 到 [index.html](/Users/zfc/code/fak111.github.io/index.html:437)。
- 当前仓库远端是 `https://github.com/fak111/fak111.github.io.git`，符合 GitHub Pages 用户站点仓库命名，说明最优部署路径应是 GitHub Pages 直出。
- 当前已有一条移动端断点样式 [index.html](/Users/zfc/code/fak111.github.io/index.html:260)，但整站尚未形成 `home / search / post / about / talk` 五个栏目。

### 1.3 技术路线

| 层级 | 技术选型 | 版本 | 说明 |
|------|----------|------|------|
| 前端 | HTML5 + CSS3 + 少量原生 JavaScript | 浏览器原生 | 保持纯前端、简单优先 |
| 路由形态 | GitHub Pages 多页面静态路由 | N/A | 不引入 SPA 路由器，降低复杂度 |
| 内容组织 | 手写静态页面 + 可复用片段结构 | N/A | 先确保上线，再决定是否抽象为模板 |
| 部署 | GitHub Pages | GitHub 托管 | 直接复用当前仓库命名优势 |
| 验收 | 真实 Chrome + 移动端视口 | Chrome Stable | 验证桌面端与移动端显示 |

### 1.4 开发原则

- 简单优先：不引入框架、打包器、CMS 或数据库。
- 静态优先：首版以多页面静态站完成信息架构落地。
- 内容优先：先解决页面结构、阅读体验和栏目组织，再扩展内容生产工具。
- 部署闭环优先：本阶段必须包含 GitHub Pages 上线和真实浏览器验收。
- 单账号验收：最终验收时必须使用登录了 `fc zhang / zhuliderb@gmail.com` 的真实 Chrome。

## 5. 技术方案

### 5.1 静态页面架构

**功能**：把当前单页内容升级为多页面博客结构，同时保持纯前端和 GitHub Pages 兼容。

**技术选型**

| 组件 | 技术 | 选型理由 |
|------|------|----------|
| 页面模板 | 原生 HTML | 最贴合当前仓库现状，无额外构建成本 |
| 公共样式 | 原生 CSS | 可以直接服务 GitHub Pages，便于响应式控制 |
| 轻交互 | 原生 JavaScript | Search 首版仅需轻量交互，不需要框架 |

**模块架构**

```text
┌────────────────────────────────────────┐
│            静态页面层                   │
├────────────────────────────────────────┤
│ Home  About  Post  Search  Talk       │
├────────────────────────────────────────┤
│         assets/styles.css              │
│         assets/app.js                  │
└────────────────────────────────────────┘
```

**实现要点**

- 保留当前页面中可复用的视觉资产，如头像资源 [profile.jpg](/Users/zfc/code/fak111.github.io/images/profile.jpg)。
- 将现有内联样式拆为公共层和页面局部层，减少后续维护成本。
- 页面间使用普通 `<a>` 链接，避免 GitHub Pages 下的客户端路由问题。

### 5.2 搜索首版方案

**功能**：在不引入后端的前提下提供可用的搜索入口。

**技术选型**

| 组件 | 技术 | 选型理由 |
|------|------|----------|
| 数据源 | 前端内置文章列表 JSON 或 JS 数组 | 静态站可直接使用 |
| 搜索逻辑 | 原生字符串匹配 | 实现成本低，满足首版要求 |
| 结果渲染 | DOM 更新 | 不引入模板引擎 |

**实现要点**

- 首版只支持标题、关键词和简短摘要匹配。
- 无结果时显示空状态，不报错、不跳空白页。
- 如果首版时间不足，保留搜索页面 UI 和输入框，逻辑最少做到前端过滤。

### 5.3 响应式方案

**功能**：满足“桌面端正常显示 + 移动端适配 + 自适应”的当前阶段硬要求。

**技术选型**

| 组件 | 技术 | 选型理由 |
|------|------|----------|
| 布局 | CSS Grid + Flexbox | 适合静态站混合布局 |
| 断点 | `1440 / 1024 / 768 / 480` | 覆盖桌面、平板、手机 |
| 媒体资源 | 自适应宽度图片 | 避免移动端溢出 |

**实现要点**

- 导航在手机端折叠为更紧凑形式，避免顶部拥挤。
- 正文宽度、行高、边距在移动端重新设定，不直接沿用桌面值。
- 所有按钮和链接都保证足够点击热区。

### 5.4 GitHub Pages 部署方案

**功能**：以当前仓库为唯一部署源，形成从本地到线上可访问网站的闭环。

**技术选型**

| 组件 | 技术 | 选型理由 |
|------|------|----------|
| 仓库 | `fak111/fak111.github.io` | 已符合用户站点仓库命名 |
| 分支 | `main` | 当前默认分支 |
| 发布方式 | GitHub Pages Branch Deploy | 对静态站最简单直接 |

**实施步骤**

1. 使用正确 GitHub 账号登录 GitHub 后台，确认仓库归属为 `fak111/fak111.github.io`。
2. 打开仓库 Settings > Pages。
3. Source 选择 `Deploy from a branch`。
4. Branch 选择 `main`，目录选择 `/ (root)`。
5. 推送上线版本到 `main`。
6. 等待 GitHub Pages 发布完成，记录线上地址 `https://fak111.github.io/`。

**上线成功判定**

- GitHub Pages 后台显示站点已发布。
- `https://fak111.github.io/` 可直接打开。
- 站点 5 个栏目至少能通过页面导航访问到对应入口。
