## 问题清单

### 严重问题 (Critical)

> 必须修复，否则影响后续文档生成

1. **[位置: 第1节“缘由”第2-12行]** 缺少结构化的产品概述
   - 现状：当前内容主要在描述做这个 blog 的个人动机和对平台的不满，没有明确产品定位、目标用户、核心价值主张，也没有把“个人博客”与“AI native 内容阵地”的关系定义清楚。
   - 建议：补齐“产品定位 / 目标用户 / 核心价值”三个小节，明确这是面向谁、解决什么问题、相对现有博客或平台工具的差异化价值。

2. **[位置: 第2节“特性”第18-37行]** 功能需求列举过于粗糙，无法直接进入设计或开发
   - 现状：文档仅提到 `post xxx.md`、`get 某篇文章`、`search/post/about/home/talk` 五个栏目，但没有定义每个功能的输入、输出、交互流程、页面行为和验收结果。
   - 建议：按功能拆分需求条目，至少为发布、获取、搜索、栏目导航分别补齐“触发条件 / 用户操作 / 系统行为 / 输出结果 / 验收标准”。

3. **[位置: 文档整体，尤其第14-37行]** 缺少非功能需求和技术约束
   - 现状：没有说明性能目标、SEO、可访问性、兼容性、安全性、部署约束，也没有定义“纯前端、简单优先”在实现上的边界。
   - 建议：新增“非功能需求”章节，明确首屏性能、移动端适配、搜索引擎可索引、静态部署约束、内容所有权与备份要求等可验证指标。

4. **[位置: 第2节“特性”第18-37行]** 缺少数据规范、异常场景和边界条件
   - 现状：`post` 命令接受什么格式的 Markdown、文章元数据字段有哪些、`get` 如何匹配文章、未命中或重复命名时如何处理、栏目中哪些内容是静态哪些是动态，这些都没有定义。
   - 建议：新增“数据规范”和“异常处理”章节，明确文章源文件格式、必要 frontmatter 字段、URL 规则、查询匹配规则、错误提示与回退行为。

### 一般问题 (Major)

> 建议修复，可提升文档质量

1. **[位置: 第2节“特性”第16行、第33-35行]** 术语和目标表述存在歧义
   - 建议：对“AI native”“一句自然语言获取”“完全仿照”“简单优先”等词给出精确定义，区分“视觉参考”“功能对标”与“逐像素复刻”，避免后续范围失控。

2. **[位置: 第2节“特性”第32-37行]** 功能优先级未标注
   - 建议：为发布、获取、搜索、about/home/talk 等能力标注 P0/P1/P2，明确首版必须交付与后续迭代项，避免实现时范围漂移。

3. **[位置: 第2节“特性”第35-37行]** 外部依赖与前提条件未说明
   - 建议：明确参考站点仅用于 UI/信息架构参考，说明是否依赖 GitHub Pages、静态生成流程、本地 CLI、搜索索引构建等前置条件。

4. **[位置: 文档整体]** 验收标准缺失，需求不可验证
   - 建议：每项核心需求补一条可测试的验收标准，例如“执行 `post demo.md` 后 1 次构建内生成文章页并可从首页访问”“输入自然语言标题后可返回唯一文章链接”等。

### 改进建议 (Minor)

> 可选优化项

1. **[位置: 第1节“缘由”第8-10行]** 当前表述带有较强情绪化和口语化表达，适合作为背景，但不适合作为正式需求描述。建议把情绪性内容压缩为“问题背景”，把需求条目改写为中性、可执行语言。

2. **[位置: 文档整体]** 当前结构为“缘由 + 特性”两段混合 prose 与 bullet，格式不利于后续生成 PRD。建议改成统一模板：背景、目标、用户、功能需求、非功能需求、数据规范、验收标准、范围边界。
